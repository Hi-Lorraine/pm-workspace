# 鼎新比价助手 — 分阶段实施方案

> 核心思路：用户看到的界面始终是对话式，后端随阶段切换。

---

## 阶段一：抢流量入口（短期，21天交付）

**目标：跑通闭环，让老板看到第一张订单**

| 项 | 方案 | 理由 |
|---|---|---|
| 产品入口 | 1药城 H5 页面，对话界面 | 手机PC都能用，零安装，流量在自己手里 |
| AI 对话 | 公司服务器上的 OpenClaw + MiniMax | 用户不用装任何东西 |
| 爬虫 | 林超的药师帮 MCP（公司服务器） | 已有能力，最快跑通 |
| 用户授权 | 用户提供账号密码，后端登录 | 有风险但短期内部验证可接受 |
| 覆盖平台 | 1药城 + 药师帮 | 先做两个，验证模式 |

**风险应对：** 先小范围内测（几十个药店），不大规模推广，控制请求频率。

---

## 阶段二：降低风险（中期，1-3个月）

**目标：降低爬虫风险，扩展平台覆盖**

| 项 | 方案 | 理由 |
|---|---|---|
| 产品入口 | 同上，H5 不变 | 用户无感切换 |
| AI 对话 | 同上 | |
| 爬虫 | 混合：能走用户端就走用户端 | PC 用户可装插件辅助，手机用户仍走服务器 |
| 用户授权 | PC 端：插件直接读登录态，不要密码 | 信任门槛降低 |
| 覆盖平台 | +九州通、小药药、199 | 林超的 cli+webview 方案上线 |

**关键动作：** 出一个浏览器插件，定位是"辅助工具"而非主产品。PC 用户装了插件后，爬虫走用户本地，合规且不怕封。手机用户继续走服务器。

---

## 阶段三：全面 OpenClaw 化（长期，3个月+）

**目标：全面 OpenClaw 化，等生态成熟**

| 项 | 方案 | 理由 |
|---|---|---|
| 产品入口 | OpenClaw 客户端（如果成熟）或继续 H5 | 老板要的最终形态 |
| AI 对话 | 用户端 OpenClaw | |
| 爬虫 | 全部 Skill 化，跑在用户端 | 零风险 |
| 用户授权 | 不需要了，用户自己登录态 | |

**前提：** OpenClaw 生态发展成熟，用户设备够好、安装门槛降低。如果到时候 OpenClaw 还不成熟，就继续用 H5 + 服务器方案，不强推。

---

## 为什么这个分法最好

1. **每个阶段用户界面都不变** — 对话式设计全程通用，不浪费
2. **短期最快** — 直接用现有能力（林超的爬虫 + 龙飞的 OpenClaw）21天跑通
3. **风险逐步降低** — 从高风险（中心化）→ 中风险（混合）→ 低风险（全用户端）
4. **和老板战略一致** — 始终是1药城流量入口，最终形态是 OpenClaw
5. **不押宝单一技术** — OpenClaw 成熟就切，不成熟就继续 H5，灵活

## 浏览器插件形态评估

### 优势

1. **天然解决反爬核心风险** — 插件运行在用户浏览器中，用自己的 IP 和登录态，等同于去中心化方案
2. **不需要用户交出账号密码** — 插件只读取页面数据，信任门槛大幅降低
3. **合规风险最低** — 用户自己的设备、IP、登录态，公司不碰第三方平台数据
4. **入口更自然** — 可以在任意平台的搜索页触发比价
5. **部署独立，迭代快** — 不需要等前端嵌入主站

### 劣势

1. **移动端缺失（最大硬伤）** — 目标用户"单体药店老板"以手机为主力工具
2. **用户安装门槛高** — 目标用户是非技术人员，装插件转化率低
3. **数据在客户端，智能层能力受限** — 跨平台数据汇总、历史数据积累受限
4. **与 OpenClaw 架构冲突** — 需要额外适配
5. **"流量入口"战略价值打折** — 用户在竞品网站就能比价，流量不一定回1药城
6. **竞品可抄** — 客户端代码暴露
7. **维护成本高** — 各平台改版需频繁更新插件

### 结论

**不建议作为主形态，但可作为阶段二的技术增强手段：**
- PC 用户装插件 → 爬虫走本地，降低风险
- 手机用户不装 → 继续走服务器
- 主产品形态始终是 1 药城 H5 对话页

## 爬虫法律风险评估

### 核心风险

| 风险 | 相关法律 | 说明 |
|------|---------|------|
| 非法获取数据 | 刑法第285条 | 绕过技术保护措施获取数据 |
| 不正当竞争 | 反不正当竞争法 | 抓取竞品数据用于商业目的 |
| 侵犯个人信息 | 个人信息保护法 | 涉及用户采购记录等 |
| 违反平台协议 | 民事纠纷 | 平台用户协议禁止自动化访问 |

### 降低风险的做法

1. **用户授权 + 知情同意** — 用户主动提供账号，明确告知用途
2. **只读不写** — 只查询价格，不代下单
3. **模拟正常用户行为** — 限制请求频率
4. **不存储竞品原始数据** — 查完即展示，不长期保存
5. **标注数据来源** — 展示价格时标注来源平台和时间
6. **长期走去中心化** — 工具跑在用户端，公司不碰平台数据

### 结论

> 去中心化 + 用户主动授权 + 只读不写 + 不存数据 = 风险最低的形态
